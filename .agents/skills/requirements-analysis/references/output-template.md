# Output Template

Use this structure by default. Omit sections that are clearly irrelevant, but keep the order stable.

## 1. 需求目标

- 用 2-4 句话概括要解决的问题、目标用户、核心价值。

## 2. 范围界定

### 2.1 本次范围

- 列出本轮需要覆盖的功能或模块。

### 2.2 暂不包含

- 列出明确不做的内容，避免范围失控。

## 3. 角色与场景

| 角色 | 核心诉求 | 关键操作 |
|------|----------|----------|
| 示例 | 示例 | 示例 |

如果没有明显的多角色结构，可以改成“核心使用场景”列表。

## 4. 功能需求拆解

按模块输出，推荐格式如下：

### 4.x 模块名

- 目标：说明模块存在的原因。
- 核心能力：
  - 能力 1
  - 能力 2
- 关键规则：
  - 规则 1
  - 规则 2
- 边界情况：
  - 异常/失败/并发/权限等情况
- 验收标准：
  - 可验证的结果 1
  - 可验证的结果 2

## 5. 非功能需求

- 性能
- 安全
- 可用性
- 可观测性
- 可扩展性
- 合规/审计

只保留真正相关的项，并给出具体要求而不是空泛描述。

## 6. 数据与接口影响

- 核心实体/对象
- 关键字段
- 状态流转
- 外部系统/第三方依赖
- API 或事件流影响

## 7. 风险与待确认项

### 7.1 风险

- 风险点 + 影响

### 7.2 待确认

- 问题
- 为什么会影响方案

## 8. 优先级与实施建议

建议使用以下分组：

- P0 / MVP：没有这些功能就无法交付
- P1：提升完整性或可用性
- P2：增强项或后续迭代

可进一步补充：

- 第一阶段
- 第二阶段
- 第三阶段

## 9. 开发任务清单

把结果转成工程可执行项，推荐按以下维度列出：

- 产品/交互
- 后端
- 前端
- 数据库
- 测试
- 运维/发布

如果用户明确要求“只分析不排期”，这一节可以缩成高层任务列表。

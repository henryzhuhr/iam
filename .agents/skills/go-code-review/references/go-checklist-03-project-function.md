# Go 代码审查清单 03: 项目组织与函数设计

本文件聚焦测试、包边界、依赖组织，以及函数参数、返回值和所有权设计。这里的核心不是追求整齐划一，而是判断代码契约是否清楚、是否容易被错误使用。

## 03. 项目组织

### 03.1 测试是否覆盖了改动真正改变的行为

- 不是所有改动都要补大量单元测试，但行为变化、bug 修复、边界条件、公开 API 和并发逻辑通常都值得验证。
- 并发相关代码在条件允许时应考虑 `go test -race`，尤其是涉及共享状态、锁和后台 goroutine 时。
- 如果仓库主要依赖集成测试、端到端测试或 golden test，也要尊重现有策略，不要机械要求“一定是单元测试”。

### 03.2 包边界和依赖方向

- 更值得关注的是包循环依赖、抽象层反转、内部类型泄漏到外部、基础设施细节渗入业务层等结构性问题。
- `go.mod`、目录层级和文件命名应以仓库现状为准；这些是项目约定，不是脱离上下文的通用裁决标准。

### 03.3 导入与别名

- 未使用导入、误导性别名、无理由的空白导入值得指出，因为它们会掩盖真实依赖。
- 别名只有在解决冲突、避免歧义或契合领域术语时才真正有价值。

## 03A. 函数

### 03A.1 参数是否表达了所有权、可变性和可选性

- 值传递还是指针传递，要结合是否需要修改、复制成本、是否含同步原语以及调用方语义判断。
- `*map`、`*[]T`、`*chan T` 并非绝对禁止，但它们比普通 `map`、`[]T`、`chan T` 更难理解，通常只有在需要重绑定、区分 nil 状态或就地修改头部结构时才有理由使用。
- 对新的请求作用域 API，`context.Context` 通常放第一个参数；如果是在实现现有接口或兼容旧 API，就应优先遵守既有契约。

### 03A.2 返回值是否清楚表达契约

- 多个返回值不是原罪，问题在于调用方是否容易理解它们的关系，以及是否会出现多个布尔值或状态码拼凑出来的隐式状态机。
- 如果返回值太多、语义太弱，优先考虑封装成具名结构体、错误类型或更清楚的领域对象。

反例：

```go
isContinue, retCode := p.processUnity()
```

更清晰的方向：

```go
retCode := p.processUnity()
```

### 03A.3 是否明确了共享还是拷贝

- 把 `map` 或 `slice` 保存到结构体、缓存或后台任务里时，要明确是“共享调用方数据”还是“防御性拷贝”。
- 同样地，返回内部维护的 `map` 或 `slice` 给调用方时，也要考虑外部修改是否会破坏不变量。

反例：

```go
func (d *Driver) SetTrips(trips []Trip) {
	d.trips = trips
}
```

在需要隔离调用方修改时更稳妥的写法：

```go
func (d *Driver) SetTrips(trips []Trip) {
	d.trips = append([]Trip(nil), trips...)
}
```

### 03A.4 组织方式是否降低了导航成本

- 构造函数、导出方法、同一 receiver 的方法如何分组，主要是可读性问题；只有当结构混乱到明显影响维护时，才值得在 review 里强调。

# Go 代码审查清单

本清单根据仓库中的 `checklist_Go.html` 整理，保留适合代码 review 场景的主干规则，便于在技能执行时快速加载。原始 HTML 更全面，本文件更偏向缺陷发现、设计风险和可操作审查项。

## 使用方式

- 优先检查【强制】项，它们更容易对应真实缺陷、回归或线上风险。
- 【建议】项主要用于可维护性、可读性和长期演进质量判断，不要把纯风格问题抬成高严重级别。
- 如果仓库本身已有更严格规范，以仓库规范为准；本清单作为兜底。
- 报告 findings 时，尽量标注是命中“原规范条款”还是“结合代码路径的推理结论”。

## 1. 风格规范

### 1.1 遵从惯例

- 【建议】遵从 Go 社区惯例，而不是引入项目私有且反直觉的写法。

### 1.2 文件长度

- 【建议】单个文件尽量不超过 800 行。超长文件通常意味着职责混杂或 review 成本过高。

### 1.3 缩进、括号、空格

- 【强制】统一使用 `gofmt` 处理缩进、括号和空格。
- 【强制】使用 tab 缩进。
- 【强制】左大括号不换行。

### 1.4 标识符命名

- 【强制】标识符遵循 `MixedCaps/mixedCaps`。
- 【强制】导出标识符大写开头，非导出标识符小写开头，避免意外暴露接口。
- 【强制】缩写保持一致，如 `IPServer`、`ipServer`。
- 【强制】标识符不重复携带包名语义。
- 【强制】包名不能与标准库重名，应简短、小写、无下划线。
- 【强制】文件名应语义明确、全小写，用下划线分词。
- 【强制】变量或常量名不要带类型前后缀。
- 【强制】错误变量名以 `err/Err` 开头，错误类型名以 `Error` 结尾。
- 【强制】receiver 命名应短、小写、与类型一致，不用 `this`、`self`。
- 【建议】未导出的全局变量或常量可用 `_` 前缀减少误用风险。

### 1.5 编码格式

- 【强制】代码文件必须使用 UTF-8。
- 【强制】换行符使用 Unix 风格。

## 2. 注释

### 2.1 通用规则

- 【建议】单行注释与内容之间留一个空格。
- 【建议】导出名字需要注释。
- 【建议】优先使用 `//`，`/* ... */` 只保留给包级文档。
- 【建议】注释满足 godoc 规范。

### 2.2 声明注释

- 【建议】包、函数、方法、结构体、接口、全局变量、类型别名都应有清晰注释。
- 【建议】块状常量或变量定义应有总说明，并为每项提供必要注释。

## 3. 错误处理

### 3.1 返回错误

- 【强制】错误统一作为最后一个返回值返回。
- 【强制】不要用内联错误码或特殊值代替 `error`。
- 【强制】遵循 fail fast，失败后尽快返回；如果不立即返回，需要注释解释原因。
- 【建议】接收 `context.Context` 的函数通常也应返回 `error`。

### 3.2 日志

- 【强制】错误不能只打日志不返回，除非当前层就是最终消费错误的边界层。
- 【建议】避免同一错误在多层重复打印，导致噪音和排障困难。

### 3.3 错误处理细节

- 【强制】不要吞掉错误。
- 【强制】不要在未知风险下继续执行后续逻辑。
- 【建议】错误要附带足够上下文，便于定位调用链和参数。

### 3.4 错误字符串

- 【强制】错误字符串应简洁、稳定，避免首字母大写和末尾标点。

### 3.5 类型断言

- 【强制】类型断言优先使用 `v, ok := ...` 形式，避免 panic。

### 3.6 panic / recover

- 【强制】`panic` 必须在当前 goroutine 内捕获。
- 【强制】自行启动的 goroutine 应在入口处考虑 `recover` 和堆栈记录策略。

## 4. 资源管理

### 4.1 主动关闭资源

- 【强制】打开的文件、网络连接、响应体、锁等资源必须被关闭或释放。
- 【强制】`defer` 关闭资源时要考虑关闭错误是否需要处理。
- 【建议】避免在大循环里无控制地 `defer`，导致资源释放被拖延。

### 4.2 内存泄漏

- 【建议】长期持有大切片的子切片或子字符串时，注意底层数组未释放问题。
- 【建议】高峰期膨胀后的 map 不会自动收缩，必要时重建。

### 4.3 并发

- 【强制】不要在闭包里直接捕获循环变量。
- 【强制】`map` 和 `slice` 不是并发安全的，多 goroutine 读写必须同步。
- 【建议】单例初始化或一次性流程可使用 `sync.Once`。

## 5. 项目组织

### 5.1 单元测试

- 【强制】项目应提供单元测试。
- 【强制】并发相关代码执行测试时应考虑 `go test -race`。
- 【建议】调试性能或定位优化问题时可结合 `-gcflags="-l"` 关闭内联。

### 5.2 模块化

- 【建议】项目应包含 `go.mod`。
- 【建议】模块名、包名和目录结构保持清晰一致。

### 5.3 包导入

- 【强制】导入分组、别名和空白导入应合理，避免无意义别名和未使用导入。

## 6. 函数

### 6.1 参数

- 【建议】同类型参数相邻摆放。
- 【建议】优先值传递，只有在需要修改、对象很大、或包含同步原语时才用指针。
- 【建议】不要给 `map`、`slice`、`chan`、`interface` 传指针。
- 【建议】优先用切片而非数组作为参数。

### 6.2 返回值

- 【强制】不要返回多个仅用于流程控制的状态位。
- 【强制】返回值个数不超过 3 个。
- 【建议】含义不清晰的多返回值可以命名，尤其当 `defer` 需要操作返回值时。

### 6.3 实现

- 【建议】把 `map` 或 `slice` 保存到结构体或对外暴露时，考虑浅拷贝和外部修改风险。

### 6.4 分组

- 【建议】构造函数放前面，导出函数放在非导出函数前，同一 receiver 的函数靠拢，工具函数放末尾。

## 7. 语言特性

### 7.1 变量声明

- 【建议】变量就近声明并缩小作用域。
- 【建议】局部变量优先用 `:=`，零值局部变量优先用 `var`。
- 【建议】相似声明使用分组。

### 7.2 语句

- 【强制】禁止变量遮蔽。
- 【强制】指针操作前确认非 nil，尤其在反序列化和解引用场景。
- 【强制】涉及外部输入的数值运算要考虑溢出、反转、截断和符号错误。

### 7.3 魔数

- 【强制】除 `0` 和 `1` 外，避免直接使用魔数。
- 【强制】重复出现两次以上的字符串字面量应考虑提取常量。

### 7.4 for / range / switch / goto

- 【建议】不要持有循环变量地址。
- 【建议】非必要不要在循环体里修改循环变量。
- 【建议】`for/switch`、`for/select` 中退出外层循环要用 label。
- 【强制】只需要 `range` 的 key 时丢弃 value；只需要 value 时把 key 写成 `_`。
- 【强制】遍历大结构体切片优先按下标访问，避免拷贝。
- 【强制】`switch` 必须有 `default`。
- 【强制】不要在 `switch` case 末尾写无意义的 `break`。
- 【强制】业务代码禁止使用 `goto`。

### 7.5 结构体

- 【强制】结构体字段排序要考虑内存对齐。
- 【强制】结构体拷贝要考虑浅拷贝风险。
- 【强制】结构体初始化使用带字段名的字面量。
- 【强制】结构体切片和 map 初始化时省略重复类型字面量。
- 【建议】结构体指针初始化优先 `&T{}` 而不是 `new(T)`。
- 【建议】初始化时忽略零值字段，除非字段名本身提供语义。
- 【建议】慎用公共结构体嵌入，避免暴露实现细节。

### 7.6 slice

- 【强制】`slice` 需合理初始化，必要时指定容量。
- 【强制】对 `slice` 做索引前确保不会越界。
- 【强制】基于旧切片生成新切片后，不要无意识地同时修改两者。
- 【强制】`copy` 前确认目标长度足够，且源目标区域不重叠。
- 【建议】空切片优先 `var s []T`。
- 【建议】零长度返回值优先用 `nil`，判断是否为空统一用 `len(s) == 0`。

### 7.7 map

- 【强制】固定元素列表直接用字面量，其他场景用 `make` 初始化。
- 【强制】可预估容量时给 `make(map, cap)`。
- 【强制】判断 key 是否存在要用 `v, ok := m[k]`，不要靠零值。
- 【强制】遍历 map 时不要写入 key。
- 【强制】禁止向 nil map 写入。

### 7.8 channel

- 【强制】channel 通常应为无缓冲或缓冲区大小为 1；其他容量需要解释。
- 【强制】channel 必须 `make` 初始化。
- 【强制】多生产者场景下要明确关闭时机，禁止重复关闭或向已关闭 channel 写数据。
- 【强制】不要假设 `select` 在多个就绪 channel 间的执行顺序。
- 【建议】收敛 channel 读写权限，生产者用 `chan<-`，消费者用 `<-chan`。

### 7.9 goroutine

- 【强制】不能泄漏 goroutine，必须能等待退出或取消。
- 【强制】协程池需要限制最大并发数。
- 【强制】不要在 `init()` 中启动 goroutine。
- 【建议】服务进程支持输出 goroutine stack dump 更利于排障。

### 7.10 context

- 【强制】`context.Context` 始终作为第一个参数，不要塞进结构体。
- 【强制】不要传 `nil` context，不确定时用 `context.TODO()`。
- 【强制】`WithTimeout`、`WithDeadline`、`WithCancel` 返回的 `cancel` 必须被调用。
- 【建议】RPC、数据库、外部 IO 等支持 context 的操作应设置超时。

### 7.11 格式化

- 【建议】自定义 `Printf` 风格函数要配合静态检查规则，避免格式串和参数错位。

### 7.12 字符串

- 【建议】关注编码、截断、拼接成本和不可变语义，避免误把字节长度当字符长度。

### 7.13 init

- 【强制】避免复杂 `init()` 逻辑，尤其不要在 `init()` 做不可控副作用。

### 7.14 interface

- 【建议】接口应尽量小，由消费者定义更合理；避免为了抽象而抽象。

### 7.15 iota

- 【建议】枚举值需要稳定语义时，谨慎依赖隐式顺序。

### 7.16 reflect

- 【建议】非必要不用反射；使用时补足类型校验和错误处理。

### 7.17 主函数退出

- 【强制】`main` 退出前应做好资源回收和优雅停机。

### 7.18 build tag

- 【强制】编译期差异处理优先用 build tag，不要在运行时分支模拟。

### 7.19 Mutex

- 【强制】`sync.Mutex` 不可复制；包含锁的结构体传值和拷贝都要非常谨慎。

### 7.20 序列化

- 【强制】序列化字段、tag 和默认值语义必须准确，避免静默不生效。
- 【建议】区分 `nil`、零值、空集合在序列化协议中的语义差异。

### 7.21 容器环境

- 【建议】运行在容器中时，关注 CPU 和内存限制，不要默认吃满宿主机资源。

### 7.22 time

- 【强制】时间运算、时区和超时处理要明确。
- 【建议】避免混用本地时间和 UTC；日志、协议字段应保持一致。

### 7.23 atomic

- 【强制】原子变量的读写必须始终原子化，禁止一边原子写一边普通读。
- 【建议】如果项目已采用 `go.uber.org/atomic`，优先沿用其类型安全封装。

### 7.24 rand / unsafe / cgo

- 【强制】安全相关随机数使用 `crypto/rand`，不要用 `math/rand`。
- 【强制】禁止随意使用 `unsafe`。
- 【强制】`cgo` 场景要避免把 Go 内存长期暴露给 C，并注意线程绑定问题。

## 8. 国际化

- 【强制】中英文文本使用对应语言的标点规范。
- 【强制】如果项目采用统一翻译函数，如 `T()`，则所有待翻译字符串必须走统一入口。
- 【强制】翻译词条不应直接使用格式化后的字符串。

## 9. 工具检查

- 【强制】每个 Go 模块都应配置并通过 `golangci-lint`。
- 【强制】至少关注以下高价值检查器：
  `bodyclose`、`errcheck`、`gocritic`、`gocyclo`、`gofmt`、`goimports`、`gomnd`、`gosec`、`gosimple`、`govet`、`ineffassign`、`nolintlint`、`revive`、`staticcheck`、`stylecheck`、`typecheck`、`unused`。
- 【建议】review 时把工具结果和代码推理分开陈述：工具证明的问题直接引用，工具未覆盖的问题补推理说明。

## Review 输出建议

- 优先输出命中【强制】条款且会造成真实 bug、竞态、资源泄漏、安全问题、协议不一致的问题。
- 命中【建议】条款但只影响风格时，除非仓库明确要求，否则降级处理。
- 如果发现仓库代码与 `checklist_Go.html` 冲突，但仓库已有明确约定，要说明冲突点，不要机械套规范。

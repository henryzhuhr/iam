# Go 代码审查清单 02: 错误处理与资源管理

本文件聚焦错误传播、日志、panic/recover、资源关闭和内存保留。这里的重点是“失败后系统会发生什么”，而不是机械套用写法模板。

## 02. 错误处理

### 02.1 错误是否被正确传播或消费

- 如果当前层不能完整处理失败，就应该把错误返回给上层，而不是只记录日志后继续执行。
- 如果当前层就是请求边界、任务入口或重试边界，可以消费错误，但要保证返回值、状态码、指标和日志一致。
- “fail fast” 不是绝对规则；关键是失败后对象状态、事务状态和可见副作用是否仍然一致。

### 02.2 错误信息是否保留了定位价值

- 跨层返回时，优先补充当前操作上下文，让调用方知道哪一步失败。
- 调用方还需要做 `errors.Is` 或 `errors.As` 判断时，优先使用 `%w` 保留原始错误链。
- 错误字符串的小写、无标点是 Go 常见惯例，但这类问题通常是低严重度，除非仓库明确要求。

### 02.3 日志与错误是否重复或互相掩盖

- 不要在中间层既记录完整错误又原样上抛，导致一条失败被多层重复打印。
- 更值得关注的是“吞错”“误报成功”“继续使用无效结果”这类行为问题，而不是日志格式本身。

### 02.4 类型断言和类型转换

- 当动态类型本来就可能变化时，使用 `v, ok := ...` 更安全。
- 当类型是由严格不变量保证的，直接断言也可能合理；review 应该检查这个不变量是否真的成立，而不是机械禁止。

### 02.5 panic / recover

- 不要把 `panic` 当作普通业务错误处理方式。
- `recover` 只在同一个 goroutine 中生效；如果后台 goroutine 的 panic 不应让整个进程崩溃，就需要在该 goroutine 入口显式决定如何恢复、记录和上报。
- 不是所有 goroutine 都必须 `recover`。关键是团队是否希望该类故障快速暴露并让进程退出，还是隔离单个任务失败。

## 02A. 资源管理

### 02A.1 资源是否在所有路径上被释放

- 文件、连接、响应体、事务、锁、ticker、timer、rows、cancel func 等资源都要看成功路径和失败路径是否成对释放。
- `defer` 是常见且正确的默认选择；只有在长循环、热点路径或资源生命周期需要更早结束时，才需要指出 `defer` 带来的延后释放问题。
- `Close`、`Commit`、`Rollback`、`Stop` 返回的错误是否重要，要按资源语义判断，不能一律忽略，也不必一律上升为高风险。

反例：

```go
for _, name := range files {
	f, err := os.Open(name)
	if err != nil {
		return err
	}
	defer f.Close()
}
```

更稳妥的写法：

```go
for _, name := range files {
	f, err := os.Open(name)
	if err != nil {
		return err
	}

	if err := handleFile(f); err != nil {
		_ = f.Close()
		return err
	}

	if err := f.Close(); err != nil {
		return err
	}
}
```

### 02A.2 内存保留与对象生命周期

- 只有当小对象会长期持有大底层数组、缓冲区或字符串时，才需要重点关注 subslice/substr 导致的内存保留问题。
- map、slice 的容量策略和回收行为会随 Go 版本与实现细节变化；除非已经看到 profile、热点路径或长期驻留对象，否则不要把“可能浪费一点内存”直接写成确定性缺陷。

反例：

```go
func head(data []byte) []byte {
	return data[:16]
}
```

在长生命周期返回值场景下更稳妥的写法：

```go
func head(data []byte) []byte {
	out := make([]byte, 16)
	copy(out, data[:16])
	return out
}
```

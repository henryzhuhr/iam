# 明确注册模型与租户归属规则

## 问题描述

当前需求文档同时存在“开放注册”“邀请注册”“一个用户只能属于一个租户”三类约束，但没有给出统一的主流程，导致用户注册完成后如何归属租户、是否允许自助创建租户、邮箱唯一性边界都不清楚。

## 背景

- REQ-002 允许开放注册和邀请注册
- 摘要和架构文档又要求用户单租户归属
- SQL 与现有用户模型采用 `(tenant_id, email)` 唯一约束，意味着邮箱跨租户可重复

如果这个问题不先定，后续会持续影响注册流程、租户初始化、邀请码机制、登录定位方式、用户导入与邀请逻辑。

## 解决方案

### 已落定结论

- [x] v1 不保留匿名开放注册，注册仅支持邀请制激活
- [x] 不允许“注册即创建租户”，用户只能加入既有租户
- [x] 邀请注册成为企业场景主流程，开放注册延期处理
- [x] 登录按租户上下文定位用户，同邮箱跨租户允许重复
- [x] 注册成功后自动登录，并进入首登完善信息页

### 需要回写的文档

- [x] 更新 `REQ-002` 的主流程、异常分支和验收标准
- [x] 更新产品摘要中的用户归属约束
- [x] 更新系统架构中的用户与租户关系说明
- [x] 更新需求总览中的优先级和依赖描述

### 推荐补充内容

- [x] 新增“注册模式”小节，明确开放注册、邀请注册、管理员创建三种模式的适用边界
- [x] 补充“邀请码 / 邀请链接 / 邀请过期”规则
- [x] 补充注册后是否自动登录、是否需要验证邮箱、是否需要完善资料
- [x] 补充涉及邮件、验证码、风控的外部依赖

### 完成标准

- [x] 能用一张流程图说明用户从注册到进入租户的完整路径
- [x] 文档能明确回答“用户属于哪个租户”“邮箱是否全局唯一”“谁有权创建租户”
- [x] TDD 与 SQL 不再需要靠猜测补齐注册行为

## 处理结果

已将 REQ-002、产品摘要、系统架构、业务流程和 TDD 统一为“邀请制注册/账号激活”模型，并新增 `user_invitations` 数据设计。

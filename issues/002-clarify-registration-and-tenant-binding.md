# 明确注册模型与租户归属规则

## 问题描述

当前需求文档同时存在“开放注册”“邀请注册”“一个用户只能属于一个租户”三类约束，但没有给出统一的主流程，导致用户注册完成后如何归属租户、是否允许自助创建租户、邮箱唯一性边界都不清楚。

## 背景

- REQ-002 允许开放注册和邀请注册
- 摘要和架构文档又要求用户单租户归属
- SQL 与现有用户模型采用 `(tenant_id, email)` 唯一约束，意味着邮箱跨租户可重复

如果这个问题不先定，后续会持续影响注册流程、租户初始化、邀请码机制、登录定位方式、用户导入与邀请逻辑。

## 解决方案

### 待决策项

- [ ] v1 是否保留开放注册；如果保留，开放注册后的用户归属到哪里
- [ ] 是否允许“注册即创建租户”，还是只能加入既有租户
- [ ] 邀请注册的主流程是否覆盖大多数企业场景，开放注册是否应降级为 P1/P2
- [ ] 登录时如果邮箱在多个租户下存在，如何定位租户
- [ ] 注册成功后的默认初始化动作是什么

### 需要回写的文档

- [ ] 更新 `REQ-002` 的主流程、异常分支和验收标准
- [ ] 更新产品摘要中的用户归属约束
- [ ] 更新系统架构中的用户与租户关系说明
- [ ] 更新需求总览中的优先级和依赖描述

### 推荐补充内容

- [ ] 新增“注册模式”小节，明确开放注册、邀请注册、管理员创建三种模式的适用边界
- [ ] 补充“邀请码 / 邀请链接 / 邀请过期”规则
- [ ] 补充注册后是否自动登录、是否需要验证邮箱、是否需要完善资料
- [ ] 补充涉及邮件、验证码、风控的外部依赖

### 完成标准

- [ ] 能用一张流程图说明用户从注册到进入租户的完整路径
- [ ] 文档能明确回答“用户属于哪个租户”“邮箱是否全局唯一”“谁有权创建租户”
- [ ] TDD 与 SQL 不再需要靠猜测补齐注册行为

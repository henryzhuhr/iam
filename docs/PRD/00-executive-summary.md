# 0. 一页摘要

> 适用于需求评审、排期和技术对齐前的快速阅读。
> 当前对应 PRD 版本：v0.2.0-draft

## 0.1 产品一句话

IAM 要解决的是 SaaS 多租户场景下重复建设认证与授权能力的问题，提供可复用的用户认证、内部服务认证、用户与租户管理、角色权限控制、MFA 和审计日志能力，帮助业务系统以较低成本接入统一的身份体系。

## 0.2 目标与边界

| 维度 | 摘要 |
|------|------|
| 业务目标 | 为 SaaS 产品提供开箱即用的 IAM 能力，降低重复开发成本 |
| 技术目标 | 支持高并发认证场景，目标 API P95 < 100ms、可用性 99.9% |
| 安全目标 | 满足企业级安全要求，支持 MFA、审计日志、Token 安全控制 |
| 使用边界 | 当前聚焦多租户 SaaS 的认证与授权，不覆盖完整组织协同或复杂审批流程 |

## 0.6 关键业务闭环

| 闭环 | 说明 |
|------|------|
| 用户登录闭环 | 用户输入账号密码，系统校验身份，按需触发 MFA，颁发 Access Token + Refresh Token，并记录登录日志 |
| 用户注册闭环 | 租户管理员发送邀请，用户通过邀请链接设置密码并激活账号，系统将用户绑定到指定租户 |
| 租户开通闭环 | 平台运营创建租户并配置配额，创建租户管理员，完成初始化后租户可用 |
| 用户入职闭环 | 租户管理员邀请用户加入租户，用户激活后登录系统，再由管理员分配角色与权限 |
| 权限生效闭环 | 管理员创建角色并分配权限，再将角色关联到用户，用户最终权限按角色并集实时生效 |

## 0.7 关键产品与技术约束

- 采用多租户模型，`tenant_id` 是数据隔离的基本边界。
- 一个用户只能属于一个租户，不同租户允许存在相同邮箱用户。
- v1 不支持匿名开放注册；用户注册必须携带租户上下文，通过邀请链接或账号激活完成。
- 认证方案采用 `JWT Access Token + Refresh Token` 双令牌模式。
- 内部服务认证采用 `AK/SK + 短期 JWT Access Token` 模式。
- 当前技术栈默认是 `Golang + Gin + grpc-go + MySQL + Redis + Kafka`，Kratos 作为后端框架备选方案保留。
- 安全能力要求覆盖密码加密、TLS、Token 黑名单、MFA、审计日志和敏感操作校验。
- **系统架构同时采用「平台 - 租户 - 应用 - 用户」业务模型和平台级 `Client` 认证模型**。

## 0.9 主要风险与待澄清项

| 项目 | 现状 | 影响 |
|------|------|------|
| 日志优先级依赖 | 登录和权限分配的验收已依赖日志，但日志需求被排到 P1 | 会导致 P0 验收与排期冲突 |
| 高级权限能力延期 | v1 已收敛为纯 RBAC + 应用范围角色，用户组权限、拒绝策略、动态组延期 | 降低权限计算复杂度 |
| 规模目标口径 | 文档中同时出现 10 万、100 万、千万级用户口径 | 会影响容量规划和架构设计 |
| 审计存储成本 | 审计日志和登录日志增加应用维度后，日志量和索引体积会上升 | 需关注留存策略与查询性能 |

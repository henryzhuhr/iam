# IAM 测试框架优化计划

> 目标：把当前 `pytest` API 测试框架的下一阶段优化方向、优先级、验收标准和依赖关系固化为一份可执行计划。
> 状态：规划中

---

## 1. 背景与当前状态

IAM 当前已经落地一套本地优先的 `pytest + httpx` 黑盒 HTTP API 测试框架。

当前可确认的状态如下：

- 默认测试路径由 `pytest` 在 session 内自动拉起 Go 服务并做健康检查
- 已具备 `tests/hooks/`、`tests/fixtures/`、`tests/helpers/`、`tests/framework/` 和 `tests/api/` 的基础分层
- 本地执行 `uv run pytest -q` 可以通过，当前结果为 `15 passed`
- 当前实际业务测试覆盖仍以 `/api/health` 为主
- 仓库里还没有把 Python 测试接入 CI 流程

这说明测试框架的骨架已经成立，下一阶段的重点不是重写，而是补上真正决定“可持续维护”的能力：质量门禁、运行配置、失败诊断、状态管理和文档规范。

## 2. 优化目标

本计划希望在不推翻现有框架形态的前提下，达成以下目标：

1. 让测试从“本地可跑”升级为“能守住回归”
2. 让测试环境切换和失败排查成本可控
3. 避免未落地抽象长期以“看起来已经支持”的状态悬空
4. 为后续认证、租户、角色、用户等有状态接口测试提前准备标准扩展位
5. 让后续维护者能从文档中直接理解下一步怎么做，而不是重复做框架判断

## 3. 明确的默认决策

为了避免后续实现者再次做高层决策，本计划固定以下默认选择：

- 继续沿用 `pytest + httpx + pytest 托管 Go 服务` 的黑盒 API 测试模型
- 不在这一阶段引入 UI/E2E 测试
- 不把 Go 单元测试和 Python API 测试强行合并为一套运行模型
- 设计背景、设计原因和 v1 落地过程继续维护在 `testing-framework-design-record.md`
- 下一阶段的优先级、验收标准和实施顺序统一维护在当前文档

## 4. 分阶段实施计划

### Phase 1. 接入 CI，建立最小质量门禁

目标：让测试从本地开发工具升级为仓库级守门机制。

固定实施内容：

- 新增独立的 CI 工作流，而不是继续混在镜像构建流程里
- 在 `main`、`dev` 的 push 和 PR 上运行 Python 测试质量检查
- CI 至少执行以下命令：
  - `uv sync --dev`
  - `uv run ruff check .`
  - `uv run pytest -q`
- 任一命令失败都应让 CI 失败

完成标准：

- 提交 Python 或测试相关改动时，仓库能自动运行 Ruff 和 pytest
- 测试失败可以在合并前暴露，而不是依赖人工本地执行

### Phase 2. 强化运行配置和失败诊断

目标：降低环境切换成本，并让失败信息直接指向问题。

固定实施内容：

- 为请求超时增加显式配置项，例如 `request timeout`
- 为启动探针增加显式配置项，例如 `health path` 或 `startup check url`
- 为启动探测轮询间隔增加显式配置项
- 在 `use existing service` 模式下，也在 session 级先做一次健康检查预检
- 启动失败时，统一输出命令、探测地址、最后一次错误和最近日志片段

完成标准：

- 修改端口、前缀或健康检查地址时，不需要改测试代码
- 命中外部服务模式时，服务不可用会在测试开始前快速失败
- 启动失败的报错信息足够支撑第一轮排查

### Phase 3. 收敛未落地抽象

目标：避免框架长期保留“预留但没有实际语义”的接口表面。

固定实施内容：

- 盘点当前已暴露但未真正启用的能力，例如 `environment`、`contract` marker、认证上下文扩展位
- 对每个预留能力做二选一处理：
  - 明确标注为“保留扩展位，当前未启用”
  - 或直接移除，待真实需求落地时再补
- 文档中显式区分“当前已实现能力”和“后续规划能力”

完成标准：

- 新维护者不会把未实现能力误读为现成功能
- 运行参数、fixture 和 marker 的语义与现状保持一致

### Phase 4. 补认证夹具与测试数据工厂

目标：为有状态业务接口测试建立统一的进入方式，避免测试文件各写各的。

固定实施内容：

- 为认证相关测试定义统一入口，优先复用 `auth_session` 扩展位
- 确立测试数据准备和清理的标准方式，例如 factory、fixture 或集中清理工具
- 约束业务测试文件不直接散落登录、造数、清理和环境拼装逻辑
- 保持 `tests/api/<domain>/` 只承载业务断言，不承载框架级样板代码

完成标准：

- 首个受保护接口测试不需要在测试文件里手写 token 拼装逻辑
- 首个有状态接口测试具备统一的数据准备与清理路径

### Phase 5. 补齐测试文档规范

目标：把框架从“少数人知道怎么用”提升为“团队能稳定扩展”。

固定实施内容：

- 新增 Test Case Writing Guide
- 新增 Testing Results Guide
- 在目录索引中明确每份文档的职责边界，避免多处重复维护

完成标准：

- 新增测试用例时，维护者知道该放哪里、怎么命名、如何断言
- 查看本地或 CI 结果时，维护者知道先看什么信息、如何判断失败归属

## 5. 验收标准

本计划完成后，应满足以下整体验收标准：

- 测试已进入 CI，且默认阻断回归
- 测试框架支持更明确的超时和健康检查配置
- 外部服务模式具备前置健康检查
- 未落地能力不会继续以“已支持”的形式暴露给使用者
- 后续业务接口测试可以通过统一认证夹具和数据工厂扩展
- 测试相关文档形成“设计记录 + 优化计划 + 使用规范”的稳定结构

## 6. 风险与依赖

实施本计划时需要注意以下依赖关系：

- CI 接入依赖仓库运行环境能稳定安装 `uv`、Python 和 Go
- 认证夹具与数据工厂的最终形态依赖后续真实业务接口的落地情况
- 如果后续引入 MySQL、Redis、Kafka 等外部依赖编排，测试启动模型可能需要继续演进，但不影响当前阶段先完成质量门禁和配置增强

## 7. 当前建议的实施顺序

除非后续需求明确改变优先级，否则统一按以下顺序实施：

1. CI 接入
2. 运行配置与失败诊断增强
3. 未落地抽象收敛
4. 认证夹具与测试数据工厂
5. 测试文档规范补齐

这个顺序的原因很简单：

- CI 能最快把当前框架转化为真实的质量门禁
- 运行配置和失败诊断会直接影响后续所有调试效率
- 抽象收敛有助于避免边开发边误用
- 认证与数据工厂应在真实业务接口进入测试前落地
- 文档规范在框架能力边界更清晰后补齐，返工成本最低

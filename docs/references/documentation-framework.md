# 软件项目文档体系框架

> 最后更新：2026-03-29
> 适用范围：IAM 项目

---

## 1. 概述

本文档从软件项目管理全流程角度，梳理一个完整项目所需的文档体系，帮助团队建立规范的文档框架。

### 1.1 文档的价值

- **知识沉淀** - 避免人员流动导致知识流失
- **沟通协作** - 降低跨角色沟通成本
- **质量保证** - 确保需求、设计、实现的一致性
- **合规审计** - 满足审计和合规要求
- **维护传承** - 降低后续维护和理解成本

## 5. 文档维护原则

### 5.1 文档分层

| 层级 | 文档类型 | 包含文档 | 读者对象 |
|------|----------|----------|----------|
| **决策层** | 项目决策文档 | BRD、项目章程、风险登记册 | 管理层/PM |
| **设计层** | 架构与设计文档 | PRD、HLD、架构设计 | 架构师/产品 |
| **执行层** | 开发与测试文档 | LLD、API 文档、测试用例 | 研发/测试 |
| **操作层** | 运维文档 | 部署手册、故障处理手册 | 运维 |
| **用户层** | 用户支持文档 | 用户手册、FAQ、发布说明 | 最终用户 |

### 5.2 文档更新时机

| 文档类型 | 更新时机 | 负责人 |
|----------|----------|--------|
| PRD | 需求变更时 | 产品经理 |
| 设计文档 | 架构调整时 | 架构师 |
| API 文档 | 接口变更时 | 开发工程师 |
| 测试用例 | 需求/设计变更时 | 测试工程师 |
| 部署手册 | 部署流程变更时 | 运维工程师 |

### 5.3 文档版本控制

- 所有文档纳入 Git 版本控制
- 文档开头标注最后更新日期
- 重大变更在文档中记录变更历史
